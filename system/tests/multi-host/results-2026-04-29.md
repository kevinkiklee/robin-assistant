## Findings

### F1. Claude Code auto-memory bypass (scenarios 2 + 6)

When the user states a preference or correction, Claude Code's built-in
auto-memory feature kicks in and writes to `~/.claude/projects/.../memory/`,
ignoring the workspace's Local Memory rule that says "all persistent
memory lives in `user-data/`".

Workarounds:
- Run `claude --bare` to disable auto-memory (also disables CLAUDE.md
  auto-discovery; would need explicit `--append-system-prompt-file=AGENTS.md`).
- Add a session-end migration step that moves auto-memory entries to
  `user-data/memory/inbox.md` (the Local Memory rule already prescribes this).
- Use `--disallowedTools "Write(~/.claude/*)"` to block the writes outright.

This is a real architectural concern, not a test bug. AGENTS.md
instructions don't override Claude Code's framework-layer auto-memory.

### F2. Headless `--print` mode skips startup sequence (scenario 5)

`claude --print` is a one-shot Q&A entry point. It doesn't run the full
multi-step session startup (read sessions.md, append session row, surface
failures, etc.). For Robin scenarios that depend on startup-sequence
side-effects, validation needs interactive mode (or a different probe).

This is documented in scenario 1's "absence of Read tool calls" finding:
Tier 1 IS in context (system prompt), but the agent doesn't perform the
ritual reads.

### F3. Trigger-phrase → protocol-fetch works (scenario 3 fix)

After re-adding the Protocols list to AGENTS.md (the slim version dropped
it), the agent correctly fetches `system/jobs/morning-briefing.md` when
the user says "morning briefing". Without the list in Tier 1, the agent
goes on a fishing expedition and never finds the protocol.
