# Protocol: Action-trust calibration

Owns the full action-state machine evolution per cycle:

1. **Capture-pipeline check** — surface a warning if `[action]` captures have been silent for 7+ days.
2. **Tally outcomes** for each `## Open` class in `action-trust.md`.
3. **Demote on correction** — first `corrected` outcome moves AUTO → ASK same cycle.
4. **Propose promotions** — eligible classes get a `<!-- proposal-id:... -->` block in `needs-your-input.md` with a 24h auto-finalize deadline.
5. **Finalize prior proposals** — proposals >24h old with no objection move the class to AUTO; objection cancels.
6. **Probation maintenance** — clear probation when the 7-day window passes with zero corrections.
7. **90-day decay** — idle AUTO classes drift back to ASK with `## Closed` reason "decay (idle 90d)".

Called from Dream Phase 12.5 (`Run system/jobs/action-trust-calibration.md inline`).
Spec: `docs/superpowers/specs/2026-05-03-action-trust-closing-the-loop-design.md`.

## Helpers

- `system/scripts/diagnostics/check-action-captures.js` — capture-pipeline scan; `scanActionCaptures(workspace)` returns `{ total, byClass, byOutcome, classesWithTrustEntry, warning7d, ... }`.
- `system/scripts/lib/needs-input.js` — `appendSection`, `clearSection`, `clearFile`, `readSections` for `needs-your-input.md`. All atomic.
- `system/scripts/capture/lib/actions/compact-summary.js` — Phase 4 step 17.5 regenerates the summary block in `policies.md` after this protocol updates the bullet body.

## Inputs

- `user-data/memory/self-improvement/action-trust.md` — the trust ledger (`## Open`, `## Closed`).
- `user-data/runtime/config/policies.md` — explicit policy. Move slugs between `## AUTO` / `## ASK` / `## NEVER` here when promoting/demoting; the compact-summary block at the top is regenerated by Dream Phase 4 step 17.5.
- `user-data/memory/streams/inbox.md` — recent `[action] <class> • <outcome> • <ref>` captures (last 30 days for tally).
- `user-data/memory/self-improvement/corrections.md` — objections to proposals (look for `[correction|origin=user] reject promotion <id>`).
- `user-data/runtime/state/needs-your-input.md` — current pending proposals + their `surfaced-at:` timestamps.

## Steps

### 1. Capture-pipeline check (sub-step)

```sh
npm run check-action-captures
```

If `warning7d:true` (use `--json` to read it programmatically), append to `needs-your-input.md` under `Action-trust capture pipeline` via `appendSection`. Banner text:

> ⚠ No `[action]` captures recorded in 7 days. Either no AUTO/ASK actions occurred (unlikely) or the capture-emission rule isn't being honored. Review `system/rules/capture.md` `### [action] tag`.

### 2. Per-class tally

For each class in `## Open`: scan `inbox.md` for `[action] <class> • <outcome> • <ref>` lines since `last_dream_at`. Increment that class's `attempts`, `successes`, `corrections`, and update `last-action`. Outcomes: `silent`/`approved` → success; `corrected` → correction; `pending` → ignore for tallying.

### 3. Demote on correction

Any class with a `corrected` outcome this cycle: move slug from the `## AUTO` list in `policies.md` to `## ASK`. Append `## Closed` entry in `action-trust.md`:

```markdown
### <class> → ASK (demoted)
- date: <YYYY-MM-DD>
- evidence: corrected outcome <ref>
```

Telemetry: append `{ event: "demotion", class, trigger: "user_correction" }` to `user-data/runtime/state/telemetry/action-trust.log` (JSONL).

### 4. Propose promotions

Eligible: ≥5 successes, 0 corrections in 30d, no privacy/dollar/legal hard-rule overlap (cross-check `system/scripts/capture/lib/actions/precheck.js`), not currently in probation, no existing un-finalized proposal for the same class.

For each eligible class, append to `needs-your-input.md` under `Action-trust promotion proposals`:

```markdown
<!-- proposal-id:YYYYMMDD-NN -->
**`<class>` → AUTO** (auto-finalize at <ISO8601 = now+24h>)
- evidence: N successes, 0 corrections, last 30 days
- last action: <YYYY-MM-DD>
- to object: append `[correction|origin=user] reject promotion <id>: <reason>` to inbox
```

Record `surfaced-at:` (= now ISO8601) in the class block in `action-trust.md` so the next cycle can compute the 24h deadline. Telemetry: `{ event: "promotion_proposed", proposal_id, class, evidence: { successes, corrections } }`.

### 5. Finalize prior proposals

For each existing proposal in `needs-your-input.md` `Action-trust promotion proposals`:

- If the matching class block in `action-trust.md` has `surfaced-at:` >24h ago AND no `[correction|origin=user] reject promotion <id>` line in `corrections.md` (or `inbox.md` since last dream): finalize.
  - Move slug from `## ASK` to `## AUTO` in `policies.md`.
  - Append `## Closed` entry: `### <class> → AUTO\n- date: <today>\n- evidence: <summary>\n- probation-until: <today + 7d>`.
  - Set `probation-until:` field on the class block in `action-trust.md`.
  - `clearSection` the proposal from `needs-your-input.md` (or rewrite the section without this proposal-id if other proposals remain).
  - Telemetry: `{ event: "promotion_finalized", proposal_id, class, probation_until }`.
- If a matching `reject` correction exists: cancel.
  - Class stays in `## ASK`.
  - Append `## Closed` entry: `### <class> → promotion rejected\n- date: <today>\n- reason: <user reason>`.
  - Remove the proposal from `needs-your-input.md`.
  - Telemetry: `{ event: "promotion_rejected", proposal_id, class, reason }`.

### 6. Probation maintenance

For each AUTO class with a `probation-until:` field: if today > probation-until and zero `corrections` since the promotion date → clear the `probation-until:` field. Telemetry: `{ event: "probation_cleared", class }`.

If a correction occurred during probation, the demotion in step 3 already handled the move; here just ensure `probation-until:` is removed from the (now-ASK) class block.

### 7. 90-day decay

For each AUTO class whose `last-action` is more than 90 days ago: move slug from `## AUTO` to `## ASK` in `policies.md`. Append `## Closed` entry: `### <class> → ASK (decay)\n- date: <today>\n- reason: idle 90d`. Telemetry: `{ event: "demotion", class, trigger: "decay_90d" }`.

## Telemetry schema

Append-only JSONL to `user-data/runtime/state/telemetry/action-trust.log`:

```jsonc
{ "ts": "...", "event": "promotion_proposed",  "proposal_id": "...", "class": "...", "evidence": {...} }
{ "ts": "...", "event": "promotion_finalized", "proposal_id": "...", "class": "...", "probation_until": "..." }
{ "ts": "...", "event": "promotion_rejected",  "proposal_id": "...", "class": "...", "reason": "..." }
{ "ts": "...", "event": "demotion",            "class": "...", "trigger": "user_correction" | "self_correction" | "decay_90d" }
{ "ts": "...", "event": "probation_cleared",   "class": "..." }
{ "ts": "...", "event": "capture_pipeline_warning", "days_silent": 14 }
```
