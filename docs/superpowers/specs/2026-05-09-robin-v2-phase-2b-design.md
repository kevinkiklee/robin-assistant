# Robin v2 Phase 2b — MCP Daemon + Agent-Facing Tools Design

**Status:** Design (pre-implementation)
**Author:** Brainstorming session, 2026-05-09
**Targets:** `robin-assistant-v2/` at HEAD `977f7f2` (tag `v6.0.0-alpha.1`).
**Scope:** Phase 2b of the v2 plan — MCP server + 10 agent-facing tools + biographer route-through + self-improvement feedback infrastructure, all built on a `robin-mcp` daemon that owns the embedded SurrealDB connection. Multi-instance Claude Code is the driving constraint.

Spec context: see `2026-05-09-robin-v2-foundation-design.md` (overall v2 plan) and `2026-05-09-robin-v2-phase-2a-design.md` (graph + biographer foundation that 2b builds on).

---

## 1. Why Phase 2b is its own spec, and what it covers

Phase 2a shipped the graph + biographer pipeline as internal-only functions. Phase 2b exposes them to agents through MCP, the committed long-term agent surface from the foundation spec.

**The driving constraint added late in 2a brainstorm: multi-instance Claude Code.** Embedded RocksDB is single-process (per Phase 1 / surrealdb skill gotcha #10). A stdio MCP server per Claude Code session would mean N processes fighting for the same DB file; the cooperative `~/.robin/.lock` from Phase 1 would serialize every operation across sessions. That's the wrong architecture for multi-instance. The fix: a single `robin-mcp` daemon owns the DB connection; MCP sessions talk to the daemon over HTTP+SSE.

Once the daemon exists, biographer should also route through it (otherwise Stop-hook subprocesses race the daemon for the DB).

### Strategic decisions locked in this brainstorm

| Decision | Choice |
|---|---|
| Phase 2b scope | Bundled (daemon + MCP + 10 tools + biographer route-through + self-improvement feedback infra, one spec, ~40–50 tasks) |
| Server lifecycle | HTTP+SSE daemon (multi-instance Claude Code requires single-writer DB) |
| Tool surface | 10 tools (broader bundle + 2 self-improvement tools: `mark_recall_used`, `record_correction`) |
| Naming | Scheme A flat, with `run_biographer` preserving Robin's domain term |
| Daemon supervision | launchd (macOS) + systemd user unit (Linux), generated by `robin mcp install` |
| Daemon auto-start | `robin mcp ensure-running` shim; for stdio transport, the existing connection process triggers it |
| Migration | User-driven (`robin migrate`); daemon refuses to start if migrations pending |
| Auth | None (binds 127.0.0.1 only, single-user local) |
| Version mismatch | Daemon refuses requests from version-skew CLI; user restarts manually |
| Self-improvement infra | New `recall_events` table auto-populated on every `recall` tool call; agent marks used hits via `mark_recall_used`; explicit corrections via `record_correction`. Feeds Phase 4 reranker + dream's heuristic loop. |

### What's NOT in 2b (deferred)

- Dream agent → 2c
- Knowledge graph + journal + threads + hot + patterns + profile → 2c
- Self-improvement loops → Phase 4
- Integrations (Gmail, Discord, etc.) → 2d
- Real-data threshold tuning → 2c
- Aliases / nickname tables → 2c
- Re-biographing already-processed events after vocabulary expansion → 2c
- Windows daemon supervision (CI is ubuntu+macos only) → unscheduled
- Streaming MCP tool responses (e.g., `recall` results streamed as biographer enriches) → unscheduled
- Authenticated MCP (multi-user) → unscheduled
- Persistent biographer queue (in-memory only in 2b; events stay `biographed_at IS NONE` and retry on next Stop hook / catchup) → 2c
- Idle-shutdown timer for the daemon → 2c

## 2. Daemon architecture

### Process model

```
┌─────────────────────────────┐         ┌─────────────────────────────┐
│  Claude Code instance #1    │         │  Claude Code instance #2    │
│    └─→ MCP client           │         │    └─→ MCP client           │
└──────────────┬──────────────┘         └──────────────┬──────────────┘
               │ HTTP+SSE                              │ HTTP+SSE
               ▼                                       ▼
         127.0.0.1:<port>  ◄── (auto-started by `robin mcp ensure-running`)
               │
   ┌───────────▼──────────────┐
   │  robin-mcp daemon        │
   │  ├─ MCP server (10 tools)│
   │  ├─ DB connection (excl) │
   │  ├─ Embedder (warm)      │
   │  └─ Biographer queue     │
   └──────────────────────────┘
               │
               ▼
        ~/.robin/db/  (RocksDB)
```

One Node process (`robin-mcp`) bound to `127.0.0.1:<port>`. Owns:
- The SurrealDB connection (`rocksdb://~/.robin/db/`).
- The embedder (lazy-loaded; held warm; unloaded after configurable idle period).
- A biographer queue (in-memory FIFO; processed one event at a time).
- The MCP HTTP+SSE server.

### Transport: verification spike picks Path A or Path B

**Verification spike at start of plan:** does Claude Code's MCP client config support an HTTP+SSE `url` transport in addition to stdio (`command` + `args`)? Check Gemini CLI's MCP integration too.

- **Path A (preferred):** Claude Code supports HTTP+SSE. `mcpServers` entry uses `url: "http://127.0.0.1:<port>/sse"`. Each Claude Code instance opens its own SSE stream. No shim. Simplest path.
- **Path B (fallback):** Only stdio supported. `mcpServers` entry uses `command: "robin", args: ["mcp", "connect"]`. The `mcp connect` subcommand is a thin stdio↔HTTP+SSE bridge: ~150 LoC, reads MCP messages from stdin, proxies to daemon via HTTP+SSE, streams results back via stdout. Adds one JSON parse + HTTP RTT per message.

Spike output is a short note (like Phase 2a's Gemini spike). Whichever path, the rest of the design stays mostly the same.

### Port management

Atomic ordering to avoid TOCTOU:

1. Acquire `~/.robin/.daemon.lock` (file lock; if `EEXIST` and PID still alive, exit `EALREADY`).
2. Bind to `127.0.0.1:0` (kernel picks free port).
3. Read the actual bound port from the listener.
4. Write `<port>` + PID + start time + version to `~/.robin/.daemon.state` (atomic via tmp+rename).
5. Begin accepting connections.

Stale-PID cleanup: `robin mcp ensure-running` reads `.daemon.lock`'s PID; if the process is dead, removes the stale lock and starts the daemon.

### Daemon startup sequence

1. Acquire `.daemon.lock`. Stale-cleanup if needed.
2. Read `.surql` migrations on disk; query `_migrations` table; if any pending, log clearly and exit non-zero. (User runs `robin migrate` manually.)
3. Open SurrealDB connection.
4. Pick port + write `.daemon.state`.
5. Start MCP HTTP+SSE server.
6. Log "daemon ready on :<port>".
7. Lazy-init embedder (don't load until first tool call needs it).

### Daemon supervision

`robin mcp install` writes:
- **macOS:** `~/Library/LaunchAgents/io.robin-assistant.mcp.plist`. Use `KeepAlive: { SuccessfulExit: false }` and `RunAtLoad: false` so the daemon is restarted on crash but not on clean exit and not on user login. **Verify launchd semantics at implementation time** — `KeepAlive`'s actual behavior across macOS versions is famously opaque. The plan must include a `launchctl print` verification step.
- **Linux:** `~/.config/systemd/user/robin-mcp.service` with `Restart=on-failure` and an `[Install] WantedBy=default.target` so the user can `systemctl --user enable robin-mcp` if they want auto-start. `loginctl enable-linger <user>` is documented as a prereq for cross-session activation.

Neither auto-starts on boot by default. Daemon spins up on first need.

### Auto-start

`robin mcp ensure-running`: small subcommand that checks `.daemon.state`, validates the port is alive, starts the daemon (detached) if dead, waits up to 5s for ready.

- Path A (HTTP transport): `mcpServers` entry includes a pre-flight that runs `ensure-running` before opening the SSE connection. Config example:

  ```json
  {
    "mcpServers": {
      "robin": {
        "url": "http://127.0.0.1:<port>/sse",
        "preflight": "robin mcp ensure-running"
      }
    }
  }
  ```

  (If Claude Code's config doesn't support `preflight`, fall back to a startup hook that runs `ensure-running` once per session. Spike documents the actual mechanism.)

- Path B (stdio shim): `robin mcp connect`'s first action is to call `ensure-running`, then start proxying.

CLI commands that need the daemon (e.g., `robin biographer-catchup`) call `ensure-running` themselves.

### Version handshake

Every CLI ↔ daemon request includes the CLI's package version in an `X-Robin-Version` header (or equivalent for stdio shim). Daemon compares to its own loaded version:

- Match → proceed.
- Mismatch → `503 version_mismatch` with both versions. CLI prints "daemon is running on `<old>`; you're on `<new>`. Restart with `robin mcp restart`." User restarts manually.

No auto-restart on mismatch (could clobber in-flight biographer work).

### Multi-session correlation

MCP supports concurrent sessions natively (per-session JSON-RPC IDs and SSE streams). `@modelcontextprotocol/sdk`'s `Server` handles this. We just need:

- **Biographer queue is global, not per-session.** Mutex around biographer execution; concurrent `run_biographer` requests coalesce — second request awaits the in-progress run's result rather than launching a parallel pass.
- **DB is global**, with SurrealDB's own transactional concurrency. No additional locking needed beyond the daemon's process-level exclusivity.

### Crash recovery

- Daemon crash mid-biographer → supervisor restart; pending events stay `biographed_at IS NONE` and get retried by the next Stop-hook signal or `robin biographer-catchup`.
- Daemon crash mid-MCP-tool-call → MCP client gets disconnect; reconnects to restarted daemon; tool retries are idempotent (Phase 2a's UPSERT-by-stable-id makes them safe).
- Stale `.daemon.state` after `kill -9` → next `ensure-running` cleans up and restarts.

### Phase 2a code that changes

| Phase 2a code | Phase 2b change |
|---|---|
| `src/hooks/stop-hook.js` | Send `POST /internal/biographer/process-pending` to daemon. Fall back to spawn-detached subprocess if daemon dead. |
| `src/cli/commands/biographer-catchup.js` | Call `ensure-running` then send request to daemon. Fall back to in-process execution if daemon refuses to start (e.g., pending migrations). |
| `src/cli/commands/migrate.js` | Add daemon-running check; refuse to migrate if `.daemon.lock` is held by a live PID. |
| `src/cli/commands/biographer-process-pending.js` | Stays for daemon-internal use AND fallback path; CLI dispatch still wired. |
| `~/.robin/.lock` (Phase 1 op-lock) | Used only by `robin migrate` and the in-process biographer fallback. Steady-state ops go through daemon. |
| `withTxRetry` helper from biographer | Stays — in-daemon biographer can still hit transient SurrealDB tx conflicts under multi-session writes. |
| `runtime:host` table | Stays. Records which adapter the daemon is using. |

## 2b. Schema additions for self-improvement feedback

New migration `0004-recall-events.surql` adds the feedback-capture table.

```surql
-- Self-improvement feedback: every recall call gets a row;
-- agent marks which hits actually informed its response.
DEFINE TABLE recall_events SCHEMAFULL TYPE NORMAL;
DEFINE FIELD query_text     ON recall_events TYPE string;
DEFINE FIELD query_vec      ON recall_events TYPE array<float>
  ASSERT array::len($value) = 384;
DEFINE FIELD hit_ids        ON recall_events TYPE array<record<events>>;
DEFINE FIELD hit_dists      ON recall_events TYPE array<float>;
DEFINE FIELD hit_used       ON recall_events TYPE array<bool>;  -- aligned with hit_ids
DEFINE FIELD session_id     ON recall_events TYPE option<string>;
DEFINE FIELD ts             ON recall_events TYPE datetime DEFAULT time::now() READONLY;
DEFINE FIELD meta           ON recall_events TYPE option<object> FLEXIBLE;
DEFINE INDEX recall_events_ts ON recall_events FIELDS ts;
DEFINE INDEX recall_events_session ON recall_events FIELDS session_id, ts;
```

**Population:** the `recall` MCP tool handler creates a row on every call. `hit_used` starts as `[false, ...]`; gets updated by the `mark_recall_used` tool. The `meta` field holds implicit signals the daemon collects (e.g., `repeat_query_within_5min: true` if the same session asked a similar query recently).

**Field details:**
- `hit_dists`: raw cosine distance from the recall query (0 = identical, 2 = opposite). More faithful to source than pre-computed similarity; trivial to convert at training time.
- `session_id`: the MCP session ID provided by `@modelcontextprotocol/sdk`'s `Server` request context. Lets multi-instance feedback attribute correctly.
- `hit_ids` as `array<record<events>>`: works in 2b because events are append-only. **Phase 2c flag:** if forget semantics ever delete events, these become dangling refs — 2c will need to either use `option<record<events>>` or coerce to strings on insert.

**Implicit-signal config:** the "similar query within 5 minutes" detector lives in `runtime:daemon.config`:

```
runtime:daemon.config = {
  ...,
  implicit_signals: {
    repeat_query_window_minutes: 5,
    repeat_query_similarity_threshold: 0.95,
    enabled: true,
  }
}
```

Tunable post-implementation without code or schema changes.

**Storage cost:** each row is ~1.5KB (384-dim float vec). At ~1000 recalls/day → ~500MB/year. Acceptable for personal use; Phase 4's training pipeline will define a retention policy when reranker training lands.

**Retention:** Phase 2b keeps all rows.

**Used by Phase 4:** `(query_vec, hit_vec, hit_features, dist) → hit_used` is the supervised signal for the learned reranker. Months of 2b/2c usage produces the training set.

## 3. MCP server + tools

### Server framework

`@modelcontextprotocol/sdk` (TypeScript MCP SDK from Anthropic, ESM). Provides `Server` class, transport implementations (`SseServerTransport`, `StdioServerTransport`), and request/response handlers. Used as-is; no custom protocol code.

Daemon binds the SSE transport to `127.0.0.1:<port>/sse`. Internal endpoint `/internal/biographer/process-pending` (POST, JSON body) for Stop-hook signals — separate from the MCP surface.

### Tool definitions

Each tool ships with: name, description, JSON Schema input, structured output, error path.

#### `recall`

```
input: {
  query: string (required, 1-500 chars),
  limit?: integer (default 10, max 50),
  source?: 'cli' | 'stop_hook' | 'manual' | 'sync' | 'biographer' | 'ingest' | 'discord' | 'migration',
  since?: ISO datetime,
  until?: ISO datetime,
  explain?: boolean (default false)
}
output: {
  recall_event_id: string (recall_events:...),  -- pass to mark_recall_used
  hits: [{ id, source, content, ts, dist, mentions: [{ entity_id, entity_name, entity_type }] }, ...],
  explain?: string
}
```

Calls Phase 2a's `recall()` then:
1. Enriches each hit with its `mentions` edges (entity name + type) — agent gets graph-aware results without a second tool call.
2. **Auto-captures feedback:** writes a `recall_events` row with the query, vector, hit IDs, hit dists, and `hit_used = [false, ...]`. Returns the row's id as `recall_event_id` so the agent can later call `mark_recall_used` to mark which hits informed its response.
3. **Implicit signals:** if a similar query (cosine sim ≥ 0.95 to any of the last 5 queries in the same session within 5 minutes) appears, sets `meta.repeat_query_within_5min: true` — implicit signal that the prior recall was unsatisfying. Used by Phase 4's reranker training.

#### `remember`

```
input: {
  content: string (required, 1-10000 chars),
  source?: string (default 'manual'),
  meta?: object,
  trigger_biographer?: boolean (default true)
}
output: { id: string }
```

Calls Phase 2a's `recordEvent()`. If `trigger_biographer: true`: enqueues the new event for biographer processing before returning.

#### `run_biographer`

```
input: {
  scope?: 'pending' | 'failed' | 'all' (default 'pending'),
  limit?: integer (default 50)
}
output: {
  processed: integer,
  failed: integer,
  failed_event_ids?: string[]
}
```

Synchronous wait; daemon processes the requested batch and returns. Concurrent calls coalesce (mutex around the biographer worker).

#### `find_entity`

```
input: {
  name: string (required),
  type?: 'person' | 'place' | 'project' | 'topic' | 'thing',
  fuzzy?: boolean (default true),
  limit?: integer (default 5)
}
output: {
  entities: [{ id, name, type, similarity?, created_at, mention_count }, ...]
}
```

`fuzzy: false` → Phase 2a Stage 1 (exact case-insensitive). `fuzzy: true` → Stage 2 (HNSW similarity). Returns top `limit` matches with similarity scores.

#### `get_entity`

```
input: { id: string (required, must be entities:...) }
output: {
  entity: {
    id, name, type, created_at, meta,
    mention_count, last_mentioned_at,
    edge_summary: { mentions: int, works_on: int, participates_in: int, co_occurs_with: int, about: int }
  }
}
```

Entity record + summary stats from edge tables.

#### `related_entities`

```
input: {
  id: string (required, entities:...),
  edge_types?: string[] (default all entity↔entity edges),
  depth?: 1 | 2 (default 1, max 2),
  limit?: integer (default 20)
}
output: {
  related: [{ entity, edge_type, strength?, distance: 1 | 2 }, ...]
}
```

SurrealDB graph traversal: `SELECT ->{edge_types}->entities FROM $id` for depth 1; recursive form `..2` for depth 2.

#### `list_episodes`

```
input: {
  source?: string,
  since?: ISO datetime,
  until?: ISO datetime,
  active_only?: boolean (default false),
  limit?: integer (default 20)
}
output: {
  episodes: [{ id, started_at, ended_at, source, summary, event_count }, ...]
}
```

`active_only: true` filters to `ended_at IS NONE`. Includes event count per episode.

#### `health`

```
input: {} (no args)
output: {
  status: 'ok' | 'degraded',
  version: string,
  uptime_seconds: number,
  db_open: boolean,
  embedder_loaded: boolean,
  pending_events: integer,
  failed_events: integer,
  active_sessions: integer,
  last_biographer_run_at: ISO datetime | null
}
```

Cheap; no DB writes.

#### `mark_recall_used` (self-improvement feedback)

```
input: {
  recall_event_id: string (required, recall_events:...),
  used_hit_ids: string[] (required; subset of the recall's returned hits)
}
output: {
  updated: integer (count of hits marked used)
}
```

Agent calls this after using `recall` results to answer a question. Sets `hit_used[i] = true` in the `recall_events` row for each hit ID that informed the response. Hits not in `used_hit_ids` stay `false` — that's the negative signal Phase 4's reranker uses.

Validation: `recall_event_id` must exist; `used_hit_ids` must all appear in the original `hit_ids` array. Out-of-set IDs are silently ignored (defensive — agent might pass IDs from a different recall by mistake).

#### `record_correction` (self-improvement feedback)

```
input: {
  content: string (required, 1-10000 chars),
  prior_response?: string (the response the user is correcting),
  meta?: object (e.g., { what_was_wrong: '...', user_preferred: '...' })
}
output: {
  id: string (events:...)
}
```

Convenience wrapper around Phase 2a's `recordEvent()` that:
- Sets `source: 'manual'` (the source enum doesn't include 'correction'; use `meta.kind: 'correction'` to mark — keeps the enum stable across phases).
- Sets `meta.kind: 'correction'`, `meta.prior_response: <text>`, plus any agent-provided meta.
- Triggers biographer **synchronously** so the correction is biographed before the tool returns. **Latency cost:** ~1–2.5s on top of the normal `remember` path. Justified because the user wants immediate effect after correcting.

Dream (Phase 2c) consumes `meta.kind: 'correction'` events to surface preference-update candidates.

**Note for Phase 2c:** dream's heuristic loop will look for clusters of corrections in `meta.kind: 'correction'` events and propose rule updates to the user.

### Biographer queue inside the daemon

Single in-memory FIFO. `run_biographer` and the Stop-hook signal both enqueue. Worker drains one event at a time using Phase 2a's `biographerProcess`. Concurrent `run_biographer` calls coalesce — second call awaits in-progress run's result.

If queue is empty and daemon is idle, embedder unloads after 10 minutes (configurable in `runtime:daemon.config`). Reloads on next call.

### Tool error handling

```js
try {
  const result = await handler(input);
  return { content: [{ type: 'json', json: result }] };
} catch (err) {
  return { isError: true, content: [{ type: 'text', text: err.message }] };
}
```

Per MCP spec, `isError: true` signals tool-level errors distinct from protocol-level errors.

## 4. AGENTS.md replacement

Phase 1's `AGENTS.md` is a placeholder. Phase 2b replaces it.

### Content (initial draft; refined in implementation)

```markdown
# AGENTS.md

You're talking to a user through Robin. Robin gives you a memory layer
through MCP tools. Use it.

## Memory tools

When the user asks "what do I know about X", "have I mentioned Y before",
"who/what is Z", or any question that requires recalling past conversations,
documents, or notes, call `recall(query=...)`. Don't guess from training data;
recall.

When you learn something noteworthy about the user, their projects, their
people, or their preferences — call `remember(content=...)`. Be discerning;
not every utterance is worth remembering. Good candidates: explicit
preferences, named projects/people first introduced, decisions, deadlines.

When you need to find a specific person/place/project/topic mentioned before,
call `find_entity(name=...)`. Use `related_entities(id=...)` to explore who
or what is connected to that entity.

When the user asks "what was I doing yesterday/last week", call
`list_episodes(since=...)`.

## When to call run_biographer

You normally don't need to. Robin runs biographer automatically after each
of your responses. Call `run_biographer` only when the user explicitly asks
"process my pending memories" or after `remember` if the user wants
immediate effect.

## Feedback (helps Robin learn)

After you use results from `recall(...)` to answer a question, call
`mark_recall_used(recall_event_id=..., used_hit_ids=[...])` with the IDs
of the hits that informed your answer. Hits that didn't help shouldn't be
in `used_hit_ids` — that's a negative signal Robin uses to improve.

When the user corrects you — "no, that's wrong", "I actually prefer X",
"the answer is Y not Z" — call `record_correction(content=..., prior_response=...)`.
This is how Robin learns your preferences and avoids repeating mistakes.
Be specific in `content`: "user prefers concise answers over detailed ones",
not just "user disagreed."

## Tone

Speak with the warmth and concision of a thoughtful friend who knows you
well. Don't be servile. Don't summarize your own actions ("I'll now call
recall..."). Just do the work and answer.
```

### Install path

`robin mcp install` writes `AGENTS.md` to the host's expected location:
- Claude Code: `~/.claude/AGENTS.md` (project-level instructions; verify path during spike).
- Gemini CLI: `~/.gemini/AGENTS.md` (verify during spike).

User can edit afterward. `robin mcp install --no-agents-md` skips it. Reinstall overwrites.

The daemon also serves AGENTS.md content as an MCP `resources/list` + `resources/read` so the agent can pull it dynamically.

## 5. Testing strategy

### Test layers

| Layer | Coverage |
|---|---|
| Unit tests on tool handlers | Each handler's JSON Schema validation + error paths, with fake DB |
| Integration tests on the daemon | Start daemon on ephemeral port, hit it with real MCP client, verify tool calls work end-to-end |
| Multi-instance tests | 2 mock MCP clients against same daemon, parallel `recall` + `remember` + `run_biographer`, assert no DB corruption |
| Lifecycle tests | `robin mcp start/stop/status/restart` exit codes + `.daemon.state` invariants |
| Migration coordination tests | Daemon refuses pending migrations; `robin migrate` refuses while daemon running |
| Connection-shim tests (Path B only) | stdio↔HTTP proxy correctness with mock MCP frames |
| Crash recovery tests | Kill daemon mid-biographer, restart, verify pending events get processed |

### Test infrastructure

- **MCP client for tests:** `@modelcontextprotocol/sdk` ships a client; integration tests use it via SSE.
- **Ephemeral port:** daemon picks port 0; tests read `.daemon.state` to discover.
- **Mock host:** stub `host.invokeLLM` for biographer tests, same pattern as Phase 2a.

### Multi-instance race tests

Critical given the architectural goal:
- 10 parallel `recall` calls — all return correct results.
- 10 parallel `remember` calls — 10 distinct events, no UPSERT collisions.
- 1 `remember` + 1 `run_biographer` racing — biographer processes the new event correctly.
- Daemon crash + restart while a tool is mid-flight — client retries succeed.

### CI implications

- GitHub Actions matrix: ubuntu-latest + macos-latest. launchd plist generation has macos-only test; systemd unit generation has linux-only test. Generated config text checked against expected.
- Daemon tests use ephemeral ports + tmp ROBIN_HOME, no persistent state across tests.

| `mark_recall_used` race when two MCP sessions both touch the same row (rare) | SurrealDB's UPDATE is atomic; last-writer-wins on the `hit_used` array. Acceptable — `mark_recall_used` is normally called once per recall_event_id by the session that received it. |
