# Arc-Assistant v2: Simplification & Portability Redesign

**Date:** 2026-04-26
**Status:** Approved design, pending implementation

## Problem

Arc-assistant v1 has grown too complex in implementation. The JavaScript code is lean (~400 lines, 2 dependencies), but the operational surface — 80+ files across 24 directories, 13 protocols with Claude Code-specific tool calls, bash coordination scripts, a Handlebars template pipeline, an override system, and a 15-signal passive capture routing table — creates friction in four areas:

1. **Maintaining it** — too many files to keep in sync, changes cascade
2. **Understanding it** — hard to hold the whole system in your head
3. **Runtime behavior** — the AI spends tokens loading, routing, and tracking instead of being helpful
4. **Feature bloat** — tracking and coordination machinery that doesn't earn its overhead

Additionally, arc-assistant is locked to Claude Code. The system should be portable across: **Cursor, Antigravity, Gemini CLI, Claude Code, Codex, and Windsurf** — with file read/write as the only assumed capability.

## Design Approach

**File-First Protocol Engine** — redesign from the ground up around the constraint that the only universal capability is file read/write. No MCP assumptions, no vendor-specific memory systems, no shell script dependencies. Protocols stay (they're valuable), but everything supporting them gets simplified.

## 1. AGENTS.md & Entry Point

### What changes

The 178-line Handlebars-generated CLAUDE.md is replaced by a static, hand-editable `AGENTS.md` (~30-40 lines).

`AGENTS.md` contains:
- Identity and hard rules (inline)
- Workspace layout (inline, points to files)
- Pointers to `startup.md`, `protocols/INDEX.md`, `capture-rules.md`, `integrations.md`

Supporting files loaded on demand:
- **`startup.md`** — session startup sequence (read handoff state, Dream check, respond)
- **`capture-rules.md`** — passive capture routing (7 buckets) + inline privacy rules
- **`integrations.md`** — generated by `arc init`, lists available capabilities per platform
- **`protocols/INDEX.md`** — protocol names and trigger phrases

### Platform pointer files

Different tools read different instruction files. `arc init` creates a platform-specific pointer alongside `AGENTS.md`:

| Platform | Pointer file | Content |
|----------|-------------|---------|
| Claude Code | `CLAUDE.md` | `Read and follow AGENTS.md for all instructions.` |
| Cursor | `.cursorrules` | `Read and follow AGENTS.md for all instructions.` |
| Gemini CLI | `GEMINI.md` | `Read and follow AGENTS.md for all instructions.` |
| Windsurf | `.windsurfrules` | `Read and follow AGENTS.md for all instructions.` |
| Codex | (reads AGENTS.md natively) | No pointer needed |
| Antigravity | TBD | Pointer file created once format is known |

### What's removed

- Handlebars template engine and generation pipeline
- Feature flags (`dream`, `multiSession`, `autoUpdateCheck`)
- `arc generate-claude-md` command
- Override system (`.local.md` shadow files)
- Tiered reading strategy (Tier 1/2/3)
- First-run detection in the instruction file (`arc init` handles setup before the AI sees the workspace)

### User-specific values

Name, timezone, and email live only in `arc.config.json`. `AGENTS.md` says "Read `arc.config.json` for user name, timezone, and email." No duplication, no sync issues on config changes.

## 2. Knowledge Store

### Consolidation: 52 files to 8

| File | Replaces | Purpose |
|------|----------|---------|
| `profile.md` | 8 files in `profile/` | Everything about the user, with section headers |
| `tasks.md` | 6 files in `todos/` | All active tasks, grouped by category |
| `knowledge.md` | 7 subdirs in `knowledge/` | Reference facts, sectioned by domain |
| `decisions.md` | `decisions/` directory | Decision log, append-only, dated entries |
| `journal.md` | `journal/` directory | Append-only dated entries |
| `self-improvement.md` | 11 files in `self-improvement/` | Sections: Corrections, Patterns, Session Handoff, Calibration |
| `inbox.md` | `inbox/inbox.md` | Quick capture, processed during Dream/maintenance |
| `artifacts/` | Same | Generated outputs (stays a directory) |

### What changes

- No INDEX.md files per directory — each file is self-contained
- Profile is one file with sections, not 8 separate files
- Self-improvement collapses from 11 files to 1 (mistakes and wins fold into journal with tags)
- Knowledge is one file with sections instead of 7 subdirectories
- If a section grows very large, it can be split out later — start consolidated

### What stays

- Append-only convention for journal, decisions, and inbox
- Dream protocol still consolidates and prunes
- Read-before-write still applies for concurrent access

## 3. Protocol Architecture

### Model-neutral language

Protocols describe **what to do**, not **which tool API to call**:
- "If email access is available (check `integrations.md`), search for recent threads" instead of "Call `ToolSearch` with `select:mcp__claude_ai_Gmail__search_threads`"
- "Read `state/session-registry.md` to check for active sessions" instead of "Run `core/coordination/register-session.sh start`"

### Integration-aware fallback pattern

Every protocol that touches external data has per-source branching:

```
## Check calendar
Read integrations.md for calendar status.
- If a native tool is listed -> use it (tool name is in integrations.md)
- If "user-provided" -> ask: "What's on your calendar today?"
- If not configured -> skip this section
```

Each integration check is independent. Meeting Prep with Calendar but no Drive skips the "Relevant docs" section. The fallback is per-source, not per-protocol.

### Protocol list (12 protocols, unchanged)

| Protocol | Integration dependency | Fallback |
|----------|----------------------|----------|
| Morning Briefing | calendar, email, weather | User pastes/summarizes |
| Weekly Review | none | -- |
| Email Triage | email | User pastes emails |
| Meeting Prep | calendar, email, storage | Per-source fallback |
| Subscription Audit | email | User provides bank statement |
| Receipt Tracking | email | User forwards/pastes receipts |
| Todo Extraction | email | User pastes email content |
| Monthly Financial | none | -- |
| Dream | none | -- |
| System Maintenance | none | -- |
| Quarterly Self-Assessment | none | -- |
| Multi-Session Coordination | none | -- |

7 of 12 protocols have zero integration dependency.

### Dream protocol simplification

Dream collapses from 237 lines / 4 phases to 2 phases:

- **Phase 1: Scan** — read the 8 core files, read `state/dream-state.md` for last Dream timestamp, identify entries in journal.md and inbox.md dated after that timestamp
- **Phase 2: Consolidate** — promote durable facts to profile.md or knowledge.md, prune resolved items from tasks.md, update self-improvement.md session handoff section, clear processed inbox entries

Key changes:
- Content-based timestamps instead of filesystem mtime
- Wall-clock budget constraints dropped (unenforceable across models)
- Dream acquires `state/locks/dream.lock` before running
- New boundary rule: Dream can read/write any of the 8 core files. Dream NEVER edits AGENTS.md, protocols/, state/locks/, or integrations.md. Dream NEVER runs external commands or makes network requests.

### Protocol trigger discoverability

Protocol names and trigger phrases appear both in `protocols/INDEX.md` and inline in `AGENTS.md` — ensures discoverability even on tools that only read the top-level instruction file.

## 4. Integration Modules

### How it works

`arc init` asks the user which platform they're using and what external access they have. It generates `integrations.md` — a flat file that protocols check at runtime.

### Generated file format

For a Claude Code user with email and calendar:
```markdown
# Integrations

Platform: claude-code

## Available
- email: gmail (native via mcp__claude_ai_Gmail__)
- calendar: google-calendar (native via mcp__claude_ai_Google_Calendar__)

## Not configured
- storage, weather, maps, health, finance, browser

## Fallback behavior
For any integration not listed above, protocols will ask the user
to provide the information directly (paste, summarize, or screenshot).
```

For a Cursor user:
```markdown
# Integrations

Platform: cursor

## Available
- email: user-provided (paste or forward)
- calendar: user-provided (paste or screenshot)

## Not configured
- storage, weather, maps, health, finance, browser

## Fallback behavior
For any integration not listed above, protocols will ask the user
to provide the information directly (paste, summarize, or screenshot).
```

### Adding integrations later

Edit `integrations.md` directly, or run `arc configure --add-integration email gmail-mcp`. No regeneration pipeline.

### What's removed

- `core/integrations-guide.md` (119 lines) — moves to `docs/` as reference
- Handlebars `{{#each integrations}}` block
- `arc.config.json` integrations tracking with status/toolPattern
- `core/skills/browser-automation.md` and `core/skills/communication-drafting.md` — dropped entirely (platform-specific, model capability handles this)

## 5. Passive Capture & Privacy

### 7 capture buckets

One bucket per writable core file:

| Signal | Destination |
|--------|------------|
| Fact about the user | `profile.md` (appropriate section) |
| Task or commitment | `tasks.md` |
| Reference knowledge | `knowledge.md` (appropriate section) |
| Decision made | `decisions.md` |
| Correction to the assistant | `self-improvement.md` -> `## Corrections` |
| Reflective observation or daily note | `journal.md` |
| Everything else | `inbox.md` |

### Capture quality filters

```
Capture bar: would a good human assistant remember this for next time?
- Only persist facts useful in a future session
- Don't capture what's already in context and won't matter next time
- Wait for recurrence or significance before storing vendors/contacts
- Never announce captures — silent competence
```

### Privacy (inline, 4 rules)

Instead of a separate 60-line privacy-scan.md with regex patterns:

```
Before writing to any file, reject content containing:
1. Full government IDs (SSN, SIN, passport numbers)
2. Full payment card or bank account numbers (last 4 digits are fine)
3. Credentials (passwords, API keys, tokens, private keys)
4. Login URLs with embedded credentials

On match: block the write, warn the user, offer to redact.
Do not log the matched content anywhere.
```

These rules are **immutable** — no override mechanism can disable them. For financial, medical, or legal facts, confirm with the user before storing.

The regex approach was more rigorous but not portable across target platforms. Defense-in-depth: the Remote Exposure Guard (immutable hard rule) prevents the workspace from ever being pushed externally. Privacy scan is a secondary defense.

### Read-before-write

Universal rule for all writes — no exceptions for appends. Cheap insurance against concurrent session conflicts.

### Batch writes

Performance hint, not a requirement: "When multiple captures arise from one message, write them in parallel if the platform supports it. Otherwise, write sequentially."

### What's removed

- `core/passive-capture.md` (separate file) — merged into `capture-rules.md`
- `core/privacy-scan.md` (separate file with regex patterns) — 4 inline rules
- `near-misses.md` logging
- 15-signal routing table
- The "where does this go?" decision tree in CLAUDE.md

## 6. Coordination

### File-based locking (replaces bash scripts)

```
state/
  sessions.md        <- active session registry
  dream-state.md     <- Dream eligibility tracking
  locks/
    <filename>.lock  <- one lock file per locked resource
```

### Session registry (`state/sessions.md`)

On startup, append a row. On noticing entries with "Last active" older than 2 hours, remove as stale. Update own "Last active" periodically (~10 tool calls or before pillar edits).

```markdown
# Active Sessions

| Session | Platform | Started | Last active | Topic |
|---------|----------|---------|-------------|-------|
| claude-code-20260426T090000Z | claude-code | 2026-04-26T09:00:00Z | 2026-04-26T09:45:00Z | email triage |
```

### Session ID format

`<platform>-<timestamp>` — e.g., `claude-code-20260426T090000Z`. Portable, no shell commands needed. Unique enough for human-driven sessions.

### Lock protocol

1. Check if `state/locks/<filename>.lock` exists
2. If it exists and timestamp < 5 minutes old — lock is held. Tell user, wait or work on something else.
3. If it exists and timestamp > 5 minutes old — stale lock. Delete and proceed.
4. If no lock exists — create `state/locks/<filename>.lock` with session ID and timestamp
5. **Confirm-after-create:** re-read the lock file immediately. If it contains a different session ID, another session won the race — delete your lock, wait, retry.
6. Read the file fresh, make your edit, then delete the lock file

Lock file format:
```
session: claude-code-20260426T090000Z
acquired: 2026-04-26T09:32:00Z
```

### Three file categories

| Category | Files | Rule |
|----------|-------|------|
| Pillar (always lock) | `AGENTS.md`, `profile.md`, `self-improvement.md` | Acquire lock before any edit |
| Mixed-use (lock for edits, safe for appends) | `tasks.md`, `knowledge.md` | Lock when editing existing content; appending new entries is safe |
| Append-only (no lock, read-before-write) | `journal.md`, `decisions.md`, `inbox.md` | Concurrent sessions append their own entries |

### Dream lock

Dream acquires `state/locks/dream.lock` before running to prevent two sessions from running Dream simultaneously.

### What's removed

- `core/coordination/lock.sh` (86 lines of bash)
- `core/coordination/register-session.sh` (129 lines of bash)
- `core/protocols/multi-session-coordination.md` (112 lines referencing bash scripts)
- Heartbeat mechanism (replaced by "Last active" timestamp)
- Process-based stale lock detection (replaced by timestamp check)

### Known limitation

File-based "check then create" locking has a theoretical race window vs. the atomic `mkdir`-based locking in v1. The confirm-after-create step closes most of the window. Residual risk is minimal with human-driven sessions (2-3 concurrent), and read-before-write prevents data loss even if both sessions write.

## 7. CLI & Scaffolding

### Commands

| Command | What it does |
|---------|-------------|
| `arc init` | Interactive setup (name, timezone, platform, integrations) -> scaffolds workspace |
| `arc configure` | Updates `arc.config.json`. `--platform` switches tools (swaps pointer file). `--add-integration` / `--remove-integration` updates `integrations.md`. |
| `arc update` | Backs up system files to `archive/`, replaces `startup.md`, `capture-rules.md`, and `protocols/` from package. Never touches user data. |
| `arc rollback` | Restores from most recent archive backup |
| `arc validate` | Checks: all expected files exist, `arc.config.json` has required fields, no stale locks |
| `arc check-update` | Version check without applying |
| `arc export` | tar.gz of workspace |
| `arc reset` | Wipes core files back to empty templates, clears `state/`. Protocols and system files untouched. |
| `arc migrate-v2` | One-time migration from v1 workspace layout (see Section 8) |
| `arc version` | Shows version |

### Removed commands

- `arc generate-claude-md` — no more generation pipeline

### Dependencies

- **commander** — stays (CLI argument parsing)
- **handlebars** — removed

Down to **1 dependency**.

### System files vs user files

`arc update` has an explicit allowlist of system-managed files:
- `startup.md`
- `capture-rules.md`
- `protocols/` (entire directory)

Everything else is user data — never touched by `arc update`. The allowlist lives in the arc-assistant package itself.

## 8. Migration from v1

### `arc migrate-v2` command

One-time operation for existing v1 workspaces.

**Step 1: Full backup**
Back up the entire workspace (minus `.git/` and `node_modules/`) to `archive/pre-v2-<date>/`.

**Step 2: Consolidate user data**

Dynamic file discovery — scan each directory for all `.md` files, use filename as section header:

| Source | Destination | Strategy |
|--------|------------|----------|
| `profile/*.md` | `profile.md` | Each file becomes a section header |
| `todos/*.md` | `tasks.md` | Each file becomes a category header |
| `knowledge/**/*.md` | `knowledge.md` | Each subdirectory/file becomes a section |
| `decisions/*.md` | `decisions.md` | Chronological append |
| `journal/*.md` | `journal.md` | Chronological append |
| `self-improvement/*.md` | `self-improvement.md` | Mapped to sections: Corrections, Patterns, Session Handoff, Calibration |
| `inbox/inbox.md` | `inbox.md` | Direct copy |

Special cases:
- `memory/short-term/last-dream.md` -> `state/dream-state.md` (transformed to new schema)
- `memory/short-term/*` (other files) -> append to `inbox.md` for manual sorting
- `memory/long-term/*` -> append to `knowledge.md` under `## Migrated (review needed)` section
- Unknown/custom files -> append to `knowledge.md` under `## Migrated (review needed)` section
- Empty template files (3-line READMEs) -> skipped

**Step 3: Create new system files**
Write `AGENTS.md`, `startup.md`, `capture-rules.md`, `integrations.md` from package templates.

**Step 4: Move protocols**
Copy rewritten v2 protocols from package to `protocols/`.

**Step 5: Create state directory**
Scaffold `state/sessions.md`, `state/dream-state.md`, `state/locks/`.

**Step 6: Create platform pointer**
Prompt for platform if not in config (default to `claude-code` for existing users). Create the appropriate pointer file.

**Step 7: Transform `arc.config.json`**
- Add `platform` field
- Remove `features` section
- Convert `integrations` from array to new format
- Keep `initialized: true`
- Bump version to `2.0.0`

**Step 8: Remove old structure**
Delete: `core/`, `memory/`, `overrides/`, `skills/`, `profile/`, `todos/`, `knowledge/`, `decisions/`, `journal/`, `self-improvement/`, `inbox/`, `share/`, `CLAUDE.md` (unless it's now the platform pointer).

**Step 9: Report**
Print summary of what was migrated. Do NOT auto-commit. Tell the user:
"Migration complete. Review the changes with `git diff` and commit when satisfied. Your pre-v2 workspace is backed up in `archive/pre-v2-<date>/`. Long-term memory files were migrated to knowledge.md — review the '## Migrated' section."

### Rollback from v2 migration

`arc rollback` detects if the most recent backup is a `pre-v2-*` snapshot and does a full workspace restore instead of just system files.
